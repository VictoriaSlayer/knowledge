# Вводная
Описывает между собой взаимодействие объектов для получения нужных результатов.
Модель взаимодействий – глобальный взгляд на поведение множества объектов, тогда как модель состояний редуцированное представление индивидуального поведения объектов.
Взаимодействия можно моделировать на разных уровнях абстрагирования:
- Use Case/вариант использования – описывает элемент функциональности предоставляемый системой её пользователям. Use Case полезны для представления неформальных требований;
- Sequence diagram/диаграммы последовательности – более детализированы, показывают сообщения которыми обмениваются объекты с течением времени(асинхронные сигналы, вызовы процедур). Sequence diagram полезна для демонстрации последовательности поведения, видимых пользователям системы;
- Activity diagram/диаграмма деятельности(активности) – содержат всю информацию и показывают поток управления между этапами вычислений. Диаграммы деятельности могут показывать не только потоки управления, но и потоки данных. С их помощью документируются этап диаграммы последовательности необходимый для реализации операции или бизнес-процесса. Один из аналогов BPMN.
- Действующее лицо (actor) – непосредственный внешний пользователь системы. Объект или множество объектов, непосредственно взаимодействующих с системой, но не являющихся ее частью. Люди, устройства, системы и т.д.

Actor может иметь только одну четко определенную цель. Объекты и классы обычно сочетают в себе несколько целей. Действующее лицо представляет только одну грань объекта в его взаимодействии с системой.
# Use Case
Различные взаимодействия действующих лиц с системой группируются в Use Case. В каждом Use Case один или несколько действующих лиц.
![[Pasted image 20241116063536.png]]
Диаграммы Use Case

Каждый Use Case описывает элемент предоставляемой системой функциональности.
![[Pasted image 20241116063557.png]]
## Требования к Use Case
1. Use Case описывают только полные транзакции.
2. Всегда необходимо определять границы системы.
3. Убедиться в атомарности акторов. Если у одного актора несколько целей, их следует разделить. Всегда связывать их с Use Case.
4. Use Case можно структурировать.
## Еще пожелания к Use Case
При составлении Use Case желательно учитывать следующие моменты:
- цели;
- результат;
- исполнители;
- предварительные и конечные условия;
- нормальный воркфлоу;
- альтернативный воркфлоу;
- примечания;

# Моделирование взаимодействий
Диаграмма деятельности показывает последовательность этапов, образующих сложный процесс, например вычислительный алгоритм или технологический процесс. В отличии от диаграммы последовательности сосредотачивает внимание на операциях(деятельности которую необходимо выполнить), а не на объектах. Показывает поток управления.
![[Pasted image 20241116063727.png]]
Назначение диаграммы – показать этапы сложного процесса и ограничения.

Диаграммы деятельности предназначены для:
- Уточнения Use Case, Sequence Diagram;
- Разработки алгоритмов;
- Диаграммы деятельности должны быть одноуровневыми;

## Отправка и получение сигналов
Кейс – док станция которую включает пользователь. Она проходит процедуру загрузки, после чего предлагает пользователю войти в систему. После ввода имени и пароля док. станция запрашивает подтверждение данных пользователя по сети. После подтверждения док. станция заканчивает процедуру загрузки.
